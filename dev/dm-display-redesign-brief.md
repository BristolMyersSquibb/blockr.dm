# Claude Design brief — redesign the `dm` model viewer

Paste this into **claude.ai/design**. Attach the `blockr.dm` repo (or the live blockr
app) during onboarding so Claude Design inherits the blockr design system — brand
colors, fonts, and the `blockr.ui` table component — automatically. A screenshot of
the current widget (see "Current state" below) is the single most useful attachment.

---

## Goal

Redesign the **data-model viewer**: the panel that displays a `dm` object inside a
blockr block. A `dm` is a *relational data model* — a set of tables connected by
primary-key / foreign-key relationships, i.e. a small database schema / ER diagram.
The viewer must let a user (1) read the schema at a glance and (2) pick one table to
preview its rows. It works functionally today but looks crude; we want a polished,
on-brand redesign.

## Audience

Data analysts and pharma/clinical-data users building data pipelines visually in
blockr. They are domain experts, not necessarily database people — the schema should
feel approachable, not like raw Graphviz output. Used inside a desktop web app
(Shiny), embedded in a block's output region (roughly 600–900 px wide, sits in a
scrollable card).

## Current state (what we have, and why it looks bad)

The viewer is a Shiny HTML widget with two stacked regions:

1. **Schema diagram** — generated by `dm::dm_draw(view_type = "keys_only")`, which
   renders a **Graphviz/DiagrammeR SVG**: boxy nodes (one per table, listing column
   names) joined by foreign-key edges. It sits in a scroll container,
   `max-height: 400px`, diagram forced to `380px`. Styling is stock Graphviz: hard
   rectangles, default fonts, cramped spacing, edges that cross awkwardly. **This is
   the ugly part.**
2. **Table preview** — appears *below* the diagram once a table is selected. A clean
   paginated HTML table (`blockr.ui::build_html_table`, 5 rows/page, click-to-sort
   columns, respects column `label` attributes). This part already looks fine and
   should be kept / lightly harmonized.

**Interaction:** the user **clicks a table node** in the diagram. JS toggles a
`dm-selected` class (currently just a 1.5px dashed blue outline,
`--blockr-blue-500 / #3b82f6`); clicking the same node again deselects. The selected
table's rows then render in region 2.

Problems to solve: the diagram looks like default Graphviz (unstyled, dated); the
selected/hover states are weak (faint dashed outline, a brightness filter on hover);
the relationship between "the thing I clicked" and "the preview that appeared" is not
visually obvious; the two regions feel disconnected; vertical space is tight.

## What to design

A cohesive **schema viewer + table preview** that fits the blockr design language.

**Content to show**
- Each table as a card/node: table name (prominent), its columns (name + ideally a
  small type hint), and a clear marker for primary-key and foreign-key columns.
- Relationship lines between tables (foreign key → primary key). Direction matters.
- A clearly selectable state for the active table.
- The row preview for the selected table (keep the existing paginated, sortable table
  — just style it to match).

**Layout & interaction (please explore options)**
- A polished node-graph / ERD treatment of the tables and their relationships — the
  modern, legible alternative to raw Graphviz. Rounded cards, generous padding, clear
  typography hierarchy (table name vs. columns), subtle keys iconography, clean edge
  routing.
- Strong, obvious **hover** and **selected** states on table nodes.
- Make the link between selected node and its preview unmistakable — e.g. preview
  appears in a side panel next to the diagram, or the selected card visibly connects
  to the preview. Side-by-side (diagram + preview) vs. stacked (current) is exactly
  the tradeoff we want you to weigh, given the ~600–900px width.
- Consider an empty/default state (nothing selected yet) that invites a click.

**Visual direction**
- Inherit the blockr design system (clean, modern, light; blue accent
  `--blockr-blue-500 / #3b82f6`). Calm and data-dense but not cluttered. Should read
  as a first-class part of a polished analytics app, not an embedded dev tool.

## Constraints (so the design is buildable)

- The schema diagram is ultimately **rendered as SVG from a graph layout** (currently
  Graphviz). A node-graph aesthetic that maps onto auto-layout edges is realistic; a
  hand-arranged bespoke layout that no auto-layout engine could reproduce is not.
- The row preview is fixed-ish: a paginated HTML `<table>` with sortable column
  headers, ~5 rows/page, column labels. Style it, don't redesign its mechanics.
- Embedded in a block card of variable but constrained width; must degrade gracefully
  when a `dm` has many tables (scroll/zoom) or wide tables (many columns).

## Deliverables

1. The redesigned viewer in **default** (nothing selected) and **table-selected**
   states.
2. Node hover + selected states called out.
3. If you propose side-by-side vs. stacked, show both so we can compare.

Start simple (one clean variation), then iterate — I'll give specific feedback on the
node card and the diagram↔preview connection.
